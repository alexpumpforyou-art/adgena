# 🏗️ AdGena — Документация по Архитектуре и Устройству Проекта

Этот документ предназначен для быстрого погружения нового разработчика (или LLM) в контекст проекта **AdGena** — SaaS-платформы для AI-генерации визуального контента (фото товаров, карточки для маркетплейсов, рекламные креативы).

---

## 🛠 Технологический стек

- **Фреймворк:** Next.js 14 (App Router)
- **Стилизация:** Vanilla CSS (CSS Modules: `*.module.css`) — *Tailwind не используется!*
- **База данных:** SQLite (через `better-sqlite3`) — БД лежит локально в файле `database.sqlite` (идеально для Railway Volume).
- **Хранилище медиа:** Cloudflare R2 (S3-совместимое)
- **AI Провайдер:** APIYI (агрегатор Stable Diffusion, Midjourney, GPT-4)
- **Платежи:** Robokassa (с поддержкой рекуррентных платежей / подписок)
- **Деплой:** Railway (сборка `output: 'standalone'`)

## ⚙️ Архитектура ключевых узлов

### 1. Авторизация и Роли (`src/lib/auth.js` & `src/lib/db.js`)
- Используется самописная JWT-авторизация (кука `adgena_session`).
- В таблице `users` есть поле `role` (`user`, `support`, `admin`).
- При логине пользователя с email, указанным в `.env` (`ADMIN_EMAILS`), ему **автоматически** назначается роль `admin`.
- Доступна OAuth авторизация через Google и Яндекс.

### 2. Генерация Контента (`src/app/api/generate/route.js`)
Самый сложный узел:
1. Юзер выбирает параметры на фронте (`/dashboard`) и грузит исходное фото.
2. Фронт шлёт запрос на `/api/generate`.
3. Бэкенд склеивает системный промпт (из `prompts.js`) с пожеланиями юзера и отправляет запрос в `apiyi.js`.
4. APIYI возвращает сырую картинку (часто огромного размера, >3MB).
5. **Важно:** Бэкенд прогоняет картинку через `sharp`, ресайзит под нужный формат и **жестко сжимает в WebP** (`quality: 98, effort: 6`). Это было сделано для решения проблемы с медленным интернетом и таймаутами в РФ.
6. Сжатый WebP заливается в Cloudflare R2 (`storage.js`).
7. У юзера списывается 1 генерация (`generations_used++`).
8. Ссылка на R2 возвращается на фронт.

### 3. Биллинг и Подписки (`src/app/api/robokassa/`)
Полностью реализован флоу для **рекуррентных списаний** (по требованиям Робокассы):
- **Checkout (`/checkout`):** Промежуточная страница, где юзер ставит галочку "Согласен на автосписания" (логируется в БД в таблицу `consents`).
- **Инициализация:** Отправляется запрос в Робокассу с параметром `Recurring=true`.
- **Webhook (`/api/robokassa/result`):** При успешной оплате юзеру выдаются лимиты и создаётся запись в таблице `subscriptions` с `initial_inv_id` (токен привязки) и датой следующего списания `next_charge_at` (+30 дней).
- **CRON (`/api/robokassa/recurring`):** Эндпоинт, который нужно дергать каждый день внешним планировщиком (cron-job.org). Он ищет подписки, у которых `next_charge_at <= now`, и отправляет дочерние платежи с `PreviousInvoiceID`.

### 4. База Данных (`src/lib/db.js`)
Таблицы:
- `users`: данные юзеров, лимиты, инфа о подписке.
- `generations`: история генераций (ссылки на "до/после", статус).
- `sessions`: сессии для JWT.
- `tickets` & `ticket_messages`: система службы поддержки.
- `subscriptions`: активные подписки.
- `consents`: логи согласий на автосписание (юридическое требование).
- `payments`: история оплат.

## 📌 Что было исправлено/сделано недавно (Важный контекст)
1. **Проблема с загрузкой фото в РФ:** Изначально R2 отдавал сырые PNG/JPEG (по 3-8 МБ). Из-за блокировок/замедлений у пользователей в РФ всё падало по таймауту. **Решение:** Внедрен `sharp` для перегона всех генераций в `WebP`. Размеры упали на 80-90% без потери качества, загрузка стала моментальной.
2. **Падения при деплое Railway:** Использование нативного модуля `better-sqlite3` заставляет Railway компилировать C++ при каждом деплое. **Решение:** Настроено время Health Check (до 120 сек) и стратегия Rolling Deploy, чтобы избежать ложных алертов о падении сервиса.
3. **Standby Mode / Standalone:** Из-за особенностей Railway `output: 'standalone'` компонент `<Image>` от Next.js (`next/image`) вызывал краши. Везде используются обычные теги `<img>`.

## 🔑 Переменные окружения (`.env`)
Критичные переменные, которые должны быть на сервере:
- `APIYI_API_KEY`, `APIYI_BASE_URL`
- `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY`, `R2_BUCKET`, `R2_PUBLIC_URL`
- `ROBOKASSA_MERCHANT_LOGIN`, `ROBOKASSA_PASSWORD1`, `ROBOKASSA_PASSWORD2`, `ROBOKASSA_TEST_MODE`
- `JWT_SECRET`, `CRON_SECRET`, `ADMIN_EMAILS`
- `GOOGLE_CLIENT_ID`, `YANDEX_CLIENT_ID` (и их секреты)
